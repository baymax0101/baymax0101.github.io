Celery 异步任务队列：定时任务与任务状态追踪

Celery FastAPI Python 异步任务 Redis
FastAPI 实战
发布日期:   2026-04-21

本文是《FastAPI 实战》系列第 6 篇，也是第一阶段的收官篇。上篇我们讲了 Redis 缓存三大问题：穿透、雪崩、击穿的原理与 Python 解法，让系统扛住了高并发读。但高并发只是开始——当用户上传一张图片要求 AI 生成文案，或者提交一份 PDF 要求全文摘要，这些动辄几秒甚至几分钟的耗时操作，如果放在同步接口里，HTTP 连接早就超时了。本文解决的核心问题是：如何把耗时任务从请求链路中剥离，让用户先拿到任务 ID，事后通过轮询或推送拿到结果？

背景与问题

你的 FastAPI 服务已经能毫秒级响应了，但有一天产品经理说：”我们要加一个功能，用户上传 100 页 PDF，系统返回全文摘要。”

你写了个同步接口，调用 LLM API 逐页处理，结果：

  • 平均耗时 3 分钟，Nginx 504 Gateway Timeout
  • 并发一高，内存爆了——每个请求都占着一个 Python 进程
  • 用户等不及刷新页面，结果生成了两份重复摘要
  • 其中一页解析失败，整个任务崩了，用户一无所获

这些问题，用异步任务队列都能解决。Celery 是 Python 生态里最成熟的方案，配合 Redis 做消息 Broker，能把”耗时操作”从 HTTP 请求中彻底解耦。

核心原理

Celery 的架构不复杂，但有几个概念容易混淆：

┌──────────────┐      发送任务      ┌──────────────┐
│   FastAPI    │ ─────────────────> │ Redis Broker │
│  (Producer)  │                    │   (消息队列)  │
└──────────────┘                    └──────────────┘
                                           │
                                           │ 拉取任务
                                           ▼
                                    ┌──────────────┐
                                    │ Celery Worker│
                                    │  (执行者)    │
                                    └──────────────┘
                                           │
                                           │ 存储结果
                                           ▼
                                    ┌──────────────┐
                                    │ Redis Backend│
                                    │   (结果存储)  │
                                    └──────────────┘
  • Producer：你的 FastAPI 应用，调用 .delay() 或 .apply_async() 把任务发送到队列
  • Broker：消息中间件，存”待执行的任务”，Redis 是最简单的选择，生产环境可以用 RabbitMQ
  • Worker：真正的执行者，一个独立的进程（或容器），从 Broker 拉任务并执行
  • Backend：结果存储，Worker 执行完后把结果写回去，方便 Producer 查询

关键点：Producer 只管发，Worker 只管执行，两者通过 Broker 解耦。FastAPI 请求只要几毫秒就能返回”任务已提交”，真正的耗时操作在后台慢慢跑。

环境准备

pip install fastapi uvicorn celery[redis] redis
  • celery[redis]：Celery 核心 + Redis 驱动
  • redis：结果存储也需要 Redis

你还需要一个运行中的 Redis 实例：

docker run -d -p 6379:6379 redis:7-alpine

项目目录结构：

app/
├── __init__.py
├── main.py              # FastAPI 入口
├── celery_app.py        # Celery 配置 ← 本文重点
├── tasks.py             # 任务定义
└── api/
    └── tasks.py         # 任务提交/查询接口

实战代码

1. Celery 应用配置

# app/celery_app.py
import os
from celery import Celery

REDIS_URL = os.getenv("REDIS_URL", "redis://localhost:6379")

celery_app = Celery(
    "worker",
    broker=REDIS_URL,
    backend=REDIS_URL,
    include=["app.tasks"],
)

# 可选配置
celery_app.conf.update(
    task_serializer="json",
    accept_content=["json"],
    result_serializer="json",
    timezone="Asia/Shanghai",
    enable_utc=True,
    # 任务结果默认保留 1 天，防止 Redis 无限增长
    result_expires=86400,
    # 任务失败自动重试 3 次
    task_default_retry_limit=3,
)

broker 负责任务分发，backend 负责结果存储，通常用同一个 Redis 实例就行。include=["app.tasks"] 告诉 Celery 去哪里找任务函数。

2. 定义异步任务

# app/tasks.py
import time
from typing import Optional

from app.celery_app import celery_app


@celery_app.task(bind=True, max_retries=3)
def analyze_pdf(self, file_path: str, page_range: Optional[str] = None) -> dict:
    """
    分析 PDF 并生成摘要。
    bind=True 让 self 指向当前任务实例，可以用 self.retry() 手动重试。
    """
    try:
        # 模拟耗时操作：PDF 解析 + LLM 调用
        time.sleep(5)

        # 模拟偶发失败（比如 LLM API 超时）
        import random
        if random.random() < 0.2:
            raise Exception("LLM API 临时不可用")

        return {
            "status": "success",
            "summary": f"《{file_path}》的摘要：这是一份关于异步任务队列的技术文档...",
            "pages_analyzed": 42,
        }

    except Exception as exc:
        # 重试间隔：指数退避（第 1 次等 60s，第 2 次等 120s，第 3 次等 240s）
        raise self.retry(exc=exc, countdown=60 * (2 ** self.request.retries))

bind=True 是关键参数，它让任务函数第一个参数 self 变成 Celery 任务实例。这样你就能：

  • 用 self.retry() 手动触发重试
  • 通过 self.request.id 获取任务 ID
  • 通过 self.request.retries 获取当前重试次数

max_retries=3 配合 countdown=60 * (2 ** retries) 实现了指数退避重试——第一次等 1 分钟，第二次等 2 分钟，第三次等 4 分钟。这比固定间隔更合理，因为下游服务（LLM API）刚挂的时候立即重试大概率还是失败，等久一点成功率更高。

3. FastAPI 中提交任务

# app/api/tasks.py
from fastapi import APIRouter, HTTPException
from celery.result import AsyncResult

from app.celery_app import celery_app
from app.tasks import analyze_pdf

router = APIRouter(prefix="/api/v1/tasks", tags=["异步任务"])


@router.post("/pdf-analysis")
async def submit_pdf_analysis(file_path: str):
    """提交 PDF 分析任务，立即返回任务 ID。"""
    task = analyze_pdf.delay(file_path)
    return {
        "task_id": task.id,
        "status": "submitted",
        "message": "任务已提交，请通过 /tasks/{task_id} 查询结果",
    }

.delay() 是 .apply_async() 的简写，它会立即返回一个 AsyncResult 对象，不会等待任务执行完成。HTTP 响应只要几毫秒，真正的 PDF 分析在后台 Worker 中慢慢跑。

4. 查询任务状态与结果

# app/api/tasks.py

@router.get("/{task_id}")
async def get_task_status(task_id: str):
    """查询任务执行状态和结果。"""
    result = AsyncResult(task_id, app=celery_app)

    if result.state == "PENDING":
        return {"task_id": task_id, "status": "pending", "progress": 0}

    elif result.state == "PROGRESS":
        return {
            "task_id": task_id,
            "status": "running",
            "progress": result.info.get("progress", 0),
        }

    elif result.state == "SUCCESS":
        return {
            "task_id": task_id,
            "status": "completed",
            "result": result.result,
        }

    elif result.state == "FAILURE":
        return {
            "task_id": task_id,
            "status": "failed",
            "error": str(result.result),
        }

    return {"task_id": task_id, "status": result.state}

Celery 任务有以下几种常见状态：

  • PENDING：任务已提交，Worker 还没开始执行
  • STARTED：Worker 开始执行（需要配置 task_track_started=True）
  • PROGRESS：自定义的中间状态，用于报告进度
  • SUCCESS：执行成功，结果在 result.result 中
  • FAILURE：执行失败，异常信息在 result.result 中
  • RETRY：正在重试中

5. 任务进度追踪（前端轮询方案）

对于耗时几分钟的任务，用户不可能干等。我们需要让 Worker 在执行过程中主动报告进度。

# app/tasks.py
import time
from app.celery_app import celery_app


@celery_app.task(bind=True)
def long_running_analysis(self, items: list[str]) -> dict:
    """批量处理任务，每完成 10% 报告一次进度。"""
    total = len(items)
    results = []

    for idx, item in enumerate(items):
        # 模拟处理单个 item
        time.sleep(0.5)
        results.append(f"Processed: {item}")

        # 每处理 10% 更新一次进度
        if idx % max(1, total // 10) == 0:
            self.update_state(
                state="PROGRESS",
                meta={"progress": int((idx + 1) / total * 100)},
            )

    return {"status": "success", "processed": total, "results": results}

self.update_state(state="PROGRESS", meta={...}) 是 Celery 的进度报告机制。FastAPI 查询接口读到 PROGRESS 状态后，把 progress 返回给前端，前端用进度条展示。

前端轮询的简单实现：

// 前端轮询示例
async function pollTaskStatus(taskId) {
    const poll = async () => {
        const res = await fetch(`/api/v1/tasks/${taskId}`);
        const data = await res.json();

        if (data.status === "completed") {
            console.log("任务完成:", data.result);
            return;
        }

        if (data.status === "failed") {
            console.error("任务失败:", data.error);
            return;
        }

        // 更新进度条
        updateProgressBar(data.progress || 0);
        setTimeout(poll, 2000);  // 每 2 秒轮询一次
    };

    poll();
}

6. 定时任务：Celery Beat

除了异步执行任务，Celery 还能做定时任务（类似 Linux 的 crontab）。

# app/celery_app.py
from celery.schedules import crontab

# 在 celery_app.conf.update() 中增加：
celery_app.conf.beat_schedule = {
    "cleanup-expired-results": {
        "task": "app.tasks.cleanup_old_results",
        "schedule": 3600,  # 每 1 小时执行一次
    },
    "daily-report": {
        "task": "app.tasks.generate_daily_report",
        "schedule": crontab(hour=9, minute=0),  # 每天上午 9 点
    },
}

# app/tasks.py

@celery_app.task
def cleanup_old_results() -> str:
    """清理超过 7 天的任务结果，防止 Redis 无限膨胀。"""
    # Celery 自带结果清理机制，但也可以自定义
    return "Cleaned up old results"


@celery_app.task
def generate_daily_report() -> dict:
    """生成每日统计报告。"""
    # 查询数据库统计昨日数据
    return {"date": "2026-04-20", "new_users": 150, "active_users": 3200}

Beat 和 Worker 是两个独立的进程：

# 启动 Worker（执行任务）
celery -A app.celery_app worker --loglevel=info

# 启动 Beat（定时调度，单独进程）
celery -A app.celery_app beat --loglevel=info

# 或者合并启动（开发环境方便，生产不推荐）
celery -A app.celery_app worker --beat --loglevel=info

7. 与 FastAPI 完整集成

# app/main.py
from fastapi import FastAPI
from app.api.tasks import router as tasks_router

app = FastAPI(title="FastAPI + Celery Demo")
app.include_router(tasks_router)


@app.get("/health")
async def health_check():
    return {"status": "ok"}

启动顺序：

# 1. 确保 Redis 在运行
docker run -d -p 6379:6379 redis:7-alpine

# 2. 启动 FastAPI
uvicorn app.main:app --reload

# 3. 启动 Celery Worker（另一个终端）
celery -A app.celery_app worker --loglevel=info

# 4. 启动 Celery Beat（如果需要定时任务）
celery -A app.celery_app beat --loglevel=info

8. 监控：Flower 可视化面板

Celery 默认没有 Web 界面，但 Flower 可以帮你监控任务执行情况。

pip install flower

# 启动 Flower
celery -A app.celery_app flower --port=5555

打开 http://localhost:5555，你能看到：

  • 正在运行的 Worker 列表
  • 任务队列长度
  • 每个任务的执行时间、重试次数、异常信息
  • 实时消费速率

生产环境建议把 Flower 放在内网，或者加一层 Basic Auth。

踩坑记录

❌ 把数据库连接直接传给 Celery 任务

# 错误做法：在 FastAPI 中创建 session，传给 Celery
db = await get_db()
analyze_pdf.delay(file_path, db)  # Celery 会序列化 db 对象，大概率报错

Celery 通过 JSON 序列化任务参数，AsyncSession 对象无法被序列化。即使你用 pickle，跨进程传递数据库连接也是灾难。

✅ 正确做法：任务函数里自己创建数据库连接。Celery Worker 是独立进程，每次执行任务时新建 session，执行完关闭。

@celery_app.task
def process_data(record_id: int):
    # 在 Worker 进程里自己建连接
    db = SessionLocal()
    try:
        record = db.query(Record).get(record_id)
        # 处理逻辑...
        db.commit()
    finally:
        db.close()

❌ 任务结果无限堆积，Redis 内存爆掉

默认配置下，Celery 会把所有任务结果永久存在 Redis 里。跑了几百万个任务后，Redis 内存告警。

# 错误做法：不配置结果过期时间
celery_app = Celery("worker", broker="redis://localhost:6379")
# result_expires 默认可能是 None，结果永久保留

✅ 正确做法：显式配置 result_expires，并配合定时清理任务。

celery_app.conf.update(
    result_expires=86400,  # 结果保留 1 天
)

如果你完全不需要查询结果（比如只是发邮件、写日志），可以干脆不存结果：

@celery_app.task(ignore_result=True)
def send_email(to: str, subject: str):
    # 发完就完，不需要查结果
    pass

❌ 重试次数设太高，把下游服务打挂

# 错误做法：无限重试 + 固定间隔
@celery_app.task(max_retries=None, default_retry_delay=10)
def call_api():
    ...

下游服务（比如 LLM API）刚挂的时候，10 个 Worker 同时疯狂重试，可能把对方从”半死不活”打成”彻底死透”。

✅ 正确做法：限制重试次数 + 指数退避 + 最大等待上限。

@celery_app.task(bind=True, max_retries=3)
def call_api(self):
    try:
        response = requests.post("https://api.llm.com/generate")
        response.raise_for_status()
        return response.json()
    except RequestException as exc:
        # 指数退避，但最多等 10 分钟
        countdown = min(60 * (2 ** self.request.retries), 600)
        raise self.retry(exc=exc, countdown=countdown)

❌ Worker 和 Beat 混在同一个进程跑生产环境

# 错误做法：生产环境这样跑
celery -A app.celery_app worker --beat --loglevel=info

--beat 会让 Worker 进程同时承担定时调度职责。如果部署了 4 个 Worker 容器，就会同时有 4 个 Beat 在调度，导致定时任务被执行 4 次。

✅ 正确做法：Beat 单独部署一个容器，Worker 横向扩展多个容器。

# docker-compose.yml 示例
version: "3.8"
services:
  web:
    build: .
    command: uvicorn app.main:app --host 0.0.0.0

  worker:
    build: .
    command: celery -A app.celery_app worker --loglevel=info
    deploy:
      replicas: 3

  beat:
    build: .
    command: celery -A app.celery_app beat --loglevel=info
    deploy:
      replicas: 1  # Beat 只能有一个实例！

  redis:
    image: redis:7-alpine

  flower:
    build: .
    command: celery -A app.celery_app flower --port=5555
    ports:
      - "5555:5555"

总结

Celery 的价值不在于”异步”本身，而在于解耦：HTTP 请求负责触发，Worker 负责执行，两者互不阻塞。配合 Redis 做 Broker，你能得到一套轻量级但生产可用的异步架构。
