AI 核心概念串讲：从 Token 到 Agent，一篇文章讲清大模型生态

无标签
发布日期:   2026-04-21

一句话摘要：用 9 个核心概念和大量生活类比，帮你从零构建对现代 AI 系统的完整认知框架，不再被 Token、Prompt、Agent 等术语绕晕。

背景与问题

过去一年，AI 领域的概念像雨后春笋一样往外冒。你刚搞懂什么叫 ChatGPT，市场上又出了 Claude、Gemini；你刚学会写 Prompt，工程师们又在讨论 Tool Calling、RAG、Agent。更别说 MCP、Harness、多模态这些看起来很高大上的词。

更头疼的是，很多教程要么太浅（只告诉你”Prompt 就是给 AI 的指令”），要么太深（一上来就给你讲 Transformer 的注意力矩阵）。你读完之后，每个词都认识，但它们之间是什么关系？数据是怎么流动的？为什么要有这么多层抽象？——依然是一团浆糊。

本文要做的，就是帮你把这团浆糊梳理成一张清晰的地图。我们不讲数学公式，不贴代码，只用生活化的类比和系统化的逻辑，带你从最底层的信息单位开始，一步步走到最上层的智能系统。读完之后，你会明白：这些概念不是孤立存在的，它们像积木一样，一块块拼出了今天你能用到的各种 AI 产品。

核心概念全景图

在深入每个概念之前，先给你一张”地图”，让你知道我们要去哪里：

Layer 5 (质量保障)  →  Harness（测试框架）
                        ↑
Layer 4 (系统整合)  →  Agent（智能体）+ Skills（技能）
                        ↑
Layer 3 (能力扩展)  →  Tool（工具调用）+ MCP（协议标准）
                        ↑
Layer 2 (交互控制)  →  Prompt（提示词）+ Context（上下文）
                        ↑
Layer 1 (信息地基)  →  LLM（大模型）+ Token（信息原子）

这五层从下到上，正好对应了 AI 系统从”能思考”到”能做事”再到”做得可靠”的完整进化路径。接下来我们逐层拆解。

第一层：地基——信息是如何被 AI 理解的

1. LLM（大语言模型）：一个读过全人类图书馆的”猜词高手”

LLM（Large Language Model） 是整个 AI 大厦的地基。但很多人对它有一个误解：以为它”理解”了语言，像人一样思考。其实不是。

更准确的说法是：LLM 是一个超级厉害的”猜词高手”。它通过阅读万亿级别的文本（相当于读完了全人类图书馆的好几轮），记住了词语之间的统计规律。当你给它一个句子开头，它做的唯一一件事就是——猜下一个词出现概率最大的是哪个。

这个过程有点像你玩”成语接龙”或者”歌词接龙”：听到”天青色等烟雨”，你脱口而出”而我在等你”，不是因为你真的在等谁，而是因为你听过太多次这首歌，大脑里的统计规律告诉你，下一个词大概率是这个。

LLM 做的就是这件事，只不过它的”大脑”有上千亿个参数，读过的书比你多几亿倍，所以猜得极其准确，准确到看起来像是”思考”。

三个关键特性你需要知道：

  • 涌现（Emergence）：当模型参数突破某个阈值（比如 600 亿左右），它会突然展现出训练时没专门教过的能力——比如数学推理、写代码、做翻译。这就像是水分子多了会形成波浪，单个水分子没有”波浪”这个属性，但足够多的水分子聚在一起就有了。涌现是今天 LLM 如此强大的核心原因。

  • 非确定性：同样的输入，LLM 每次输出可能略有不同。这是因为它在”猜词”时会加入一点随机性——就像你有时候也会犹豫”这个词好还是那个词好”。这个特性让 AI 有创造力，但也意味着它可能”胡说八道”。

  • 知识截止：LLM 的训练数据有截止日期（比如 GPT-4 的知识截止到 2024 年初）。它不知道今天的新闻，不知道你公司内部的文档，甚至不知道昨天发生了什么事。这个缺陷很重要，因为它直接催生了后面要讲到的 Tool 和 RAG 技术。

2. Token：AI 世界里的”信息原子”

如果你以为 AI 像人类一样一个字一个字地阅读，那就大错特错了。在 LLM 眼中，文字不是”字”，而是 Token。

Token 是什么？ 你可以把它理解为 AI 处理信息的最小单位。它不一定等于一个字或一个单词，而是模型在训练时自己”发明”的一种文本切分方式。

举个例子：

  • 英文中，unbelievable 可能被切成 un + believ + able 三个 Token
  • 中文中，”人工智能”可能被切成”人工”+”智能”两个 Token，也可能是一个 Token
  • 一个标点符号、一个空格，都可能单独是一个 Token

为什么 Token 很重要？

第一，它决定了 AI 的”视野”。模型一次能处理的 Token 数量是有限的（比如 4K、32K、128K），这叫”上下文窗口”。如果你的文档太长，超过了这个窗口，AI 就会”看不见”前面的内容——就像你背一篇太长的课文，背到后面忘了前面。

第二，它是计费的单位。你调用 OpenAI API 时，账单上写的不是”多少个字”，而是”多少 Token”。一般来说，1 个汉字 ≈ 11.5 个 Token，1 个英文单词 ≈ 12 个 Token。所以你写中文 Prompt 可能比英文更”划算”一点。

第三，它影响了模型对语言的理解。因为 Token 的切分方式是在训练中自动学习的，不同语言、不同领域的 Token 效率不一样。这也是为什么同样的模型，处理中文和英文的效果可能有差异。

第二层：交互——如何与这个”猜词高手”对话

有了地基，接下来我们要解决一个问题：怎么让 LLM 按照我们的意愿输出？这就涉及到两个核心概念：Context 和 Prompt。

3. Context（上下文）：AI 的”临时记忆”

Context 指的是模型在生成当前回复时，能够”看到”的所有信息。它包括：

  • 你之前说的所有话
  • 它自己之前回复的所有话
  • 你上传的文件内容
  • 系统预设的指令（比如”你是一个乐于助人的助手”）

你可以把 Context 理解为 AI 的临时工作记忆。它在回答你这个问题的时候，会把所有这些信息摊在桌面上参考。但一旦开启新对话，这张桌子就被清空了，它什么都不记得——这就是为什么你新开一个聊天窗口，AI 会”失忆”。

上下文窗口（Context Window） 是 Context 的物理上限。如果一个模型的窗口是 128K Token，而你们的对话加起来超过了这个长度，模型就会开始”遗忘”最早的内容。不是删除，而是它在生成每个词时，注意力会更多地放在最近的 Token 上，远处的 Token 影响越来越小。

这里有一个很巧妙的机制叫注意力机制（Attention）。你可以想象成：当你读一封很长的邮件，你不会对每个字都给予同样的关注——你会重点看人名、日期、关键结论。AI 也是如此，它在处理 Context 时，会给不同的 Token 分配不同的”关注度权重”，从而聚焦于当前最相关的信息。

一个实用的技巧：如果你要让 AI 处理长文档，最好把最重要的指令放在开头和结尾。因为中间的内容最容易被”注意力”忽略——这就像是老师批改作文，开头和结尾印象最深刻。

4. Prompt（提示词）：操控”猜词高手”的遥控器

如果说 LLM 是一个猜词高手，那 Prompt 就是你给它的”题目描述”。同样的猜词能力，题目描述得好，它就能写出莎士比亚级别的散文；描述得不好，它就可能胡言乱语。

Prompt 的本质，是在配置模型的生成状态。你可以把它理解为一种”软性编程”——不用写代码，而是用自然语言来设定规则、提供背景、指定格式。

好的 Prompt 通常包含这几个部分：

  1. 角色设定：”你是一位资深 Python 后端工程师”——这会让模型激活与 Python、后端相关的知识区域。
  2. 任务描述：清晰地说出你要它做什么。
  3. 输入数据：提供需要处理的内容。
  4. 输出格式：告诉它你希望以什么形式输出（JSON、Markdown、表格、代码块等）。
  5. 约束条件：”不要超过 200 字”、”不要使用专业术语”等。

思维链（Chain of Thought, CoT） 是 Prompt 工程中最重要技巧之一。它的核心思想很简单：在 Prompt 里加上一句”请一步步思考”或者”请展示你的推理过程”。

为什么这很有效？还记得吗，LLM 是一个”猜词高手”。当你要求它”一步步思考”时，它会把中间的推理步骤也生成出来，写在 Context 里。这些中间步骤就像”路标”，引导着后续的生成走向正确的方向。实验证明，这个简单的技巧能让数学题的准确率提升 30% 以上。

一个常见的误区：很多人觉得 Prompt 写得越长越详细越好。其实不是。过于冗长的 Prompt 会挤占 Context 空间，而且模型容易”抓不住重点”。好的 Prompt 是精确、结构化、有层次的，而不是堆砌关键词。

第三层：赋能——让 AI 从”空谈”到”实干”

LLM 有两个致命的先天缺陷：知识滞后（不知道训练之后发生的事）和计算不可靠（数学题可能算错）。Tool（工具调用）就是为了弥补这两个缺陷而生的。

5. Tool（工具调用）：给 AI 装上”手脚”

Tool 允许 LLM 在需要时，调用外部程序来完成任务。

想象这样一个场景：你问 AI”今天北京天气怎么样”。LLM 本身不知道今天天气——它的知识有截止日期。但如果我们给它配备一个”查询天气”的工具，事情就不一样了：

  1. 你问：”今天北京天气怎么样？”
  2. LLM 分析后意识到：我需要实时天气数据，但我不知道。
  3. LLM 生成一个工具调用请求：”请调用 weather_query 工具，参数 city=北京”。
  4. 程序执行这个工具，拿到天气数据：”晴，25°C”。
  5. 数据被塞回 Context，LLM 基于这个真实数据生成回答：”今天北京天气晴朗，气温 25 度，适合出门。”

这个流程的精妙之处在于：LLM 只负责”决定要不要调用工具”和”理解调用结果”，实际的计算和数据获取由外部程序完成。这就好比一个 CEO 不需要自己会写代码，但他要知道什么时候该叫程序员来帮忙。

常见的 Tool 类型包括：

  • 搜索工具：让 AI 能查最新资讯（弥补知识滞后）
  • 计算工具：让 AI 调用 Python/JavaScript 执行精确计算（弥补计算不可靠）
  • 数据库查询：让 AI 能查企业内部数据
  • API 调用：让 AI 能操作第三方服务（发邮件、订机票、创建日程）

Tool 的出现，是 AI 从”聊天玩具”走向”生产工具”的关键分水岭。

6. MCP（模型上下文协议）：AI 世界的”USB 接口”

Tool 很强大，但很快就出现了一个问题：每家厂商的工具接口都不一样。

OpenAI 有自己的 Function Calling 格式，Anthropic 有另一种格式，Google 又是一种。如果你想让 AI 连接你的 GitHub、Slack、Notion，每个平台你都要写一套适配代码。这就像十年前，每个手机品牌都有自己的充电接口，出差得带一堆充电器。

MCP（Model Context Protocol，模型上下文协议） 就是来解决这个问题的。它是由 Anthropic 发起的一个开放标准，相当于 AI 界的 USB-C 接口。

MCP 的核心思想是：只要你的数据源或工具遵循 MCP 协议，任何支持 MCP 的 AI 客户端都能直接连接，无需额外适配。

打个比方：

  • 以前：你想让 AI 读你的 GitHub 仓库 → 写 GitHub API 适配代码 → 写 OpenAI Function 格式 → 调试
  • 现在：GitHub 提供一个 MCP Server → 你的 AI 客户端直接连接 → 开箱即用

MCP 标准化了三个核心能力：

  1. Resources（资源）：AI 能读取的数据（文件、数据库记录、API 响应）
  2. Tools（工具）：AI 能调用的功能（发送消息、创建文件、执行命令）
  3. Prompts（提示模板）：预定义的 Prompt 模板，方便复用

MCP 的出现意味着 AI 工具生态正在从”野蛮生长”走向”标准化”。对开发者来说，这意味着接入成本大幅降低；对用户来说，这意味着 AI 能做的事会爆发式增长。

第四层：系统——从”回答问题”到”完成任务”

当 LLM 配上 Tool，再套上一个”思考-行动-观察”的循环，它就进化成了 Agent。

7. Agent（智能体）：一个能自主干活的”数字员工”

Agent 是 2024 年以来 AI 领域最火的概念之一。但很多人对它的理解有偏差：Agent 不是某种新技术，而是一种新的系统架构——让 LLM 在一个循环中自主决策、调用工具、观察结果、调整策略。

传统的 AI 交互是”一问一答”：你问一个问题，它答一次，对话结束。

Agent 的交互是”目标驱动”：你给它一个模糊的目标，它会自己拆解成多个步骤，一步步执行，直到完成。

举个例子。你说：”帮我调研一下近三年内新能源汽车行业的发展趋势，写一份简要报告。”

一个 Agent 的处理流程可能是这样的：

Step 1: 规划 → 拆解目标：需要查政策、查销量数据、查技术趋势、查主要玩家
Step 2: 行动 → 调用搜索工具，查"2023-2025 新能源汽车政策"
Step 3: 观察 → 阅读搜索结果，提取关键信息
Step 4: 思考 → "政策部分有了，还需要销量数据"
Step 5: 行动 → 调用搜索工具，查"2023-2025 新能源汽车销量统计"
Step 6: 观察 → 整理销量趋势
Step 7: 思考 → "还需要看看技术方向，比如固态电池、自动驾驶"
Step 8: 行动 → 调用搜索工具...
...（循环继续）
Step N: 整合 → 把所有信息组织成报告格式，输出给用户

注意这个过程中的关键点：LLM 不仅是在生成文本，它还在做决策——决定下一步该搜索什么、什么时候信息够了可以写报告、报告的结构怎么安排。

自反思（Self-reflection） 是高级 Agent 的重要能力。如果某个工具调用失败了（比如搜索引擎返回了错误结果），Agent 会意识到这一点，然后尝试修正策略——换关键词重新搜索，或者换一个工具。这就像是一个有工作经验的员工，遇到问题不会傻等，而是自己想办法解决。

当然，今天的 Agent 还远没有达到”完全自主”的程度。它们经常会在循环里”迷路”——反复调用同一个工具、陷入死循环、或者产生不切实际的计划。这也是为什么 Agent 目前更适合做有明确边界、可验证步骤的任务，而不是完全开放的”你看着办”。

8. Agent Skills（技能）：让”数字员工”专业化

一个通用的 Agent 什么都能做一点，但什么都不精。如果要让它在特定领域表现更好，就需要给它配置 Skills（技能）。

Skill 本质上是一套预定义的”工具 + Prompt 模板 + 执行流程”的组合。你可以把它理解为岗位培训手册：一个财务分析师需要掌握的技能（读报表、算指标、写分析），和一个前端工程师需要的技能（读代码、调样式、写组件）完全不同。

举个例子：

  • “数据分析 Skill”：包含读取 CSV 的工具、数据清洗的 Prompt 模板、常用统计方法的调用流程
  • “代码审查 Skill”：包含静态分析工具、代码风格检查规则、常见 Bug 模式的识别 Prompt
  • “客户服务 Skill”：包含查询订单状态的工具、退换货政策知识库、安抚客户情绪的话术模板

通过给 Agent 配置不同的 Skills，我们可以快速”培养”出针对不同场景的专业化智能体，而不需要重新训练底层模型。这大大降低了定制 AI 系统的成本和门槛。

第五层：质量——如何让 AI 系统靠谱地运行

好了，现在你有了一套能思考、能调用工具、能自主执行的 AI 系统。但还有一个大问题：它靠谱吗？

别忘了 LLM 的核心特性——非确定性。它可能这次答对了，下次就错了；可能给 A 用户一个答案，给 B 用户另一个答案。在生产环境中，这种不确定性是致命的。

9. Harness（测试框架）：AI 系统的”质检部”

Harness 指的是对 AI 系统进行评估、测试和监控的一整套机制。它是确保 AI 从”实验室玩具”变成”工业级产品”的关键。

Harness 包含三个层面的工作：

1. 基准测试（Benchmark）

就像学生要考试一样，AI 也要考试。Benchmark 就是 AI 界的”标准化考试”。常见的 Benchmark 包括：

  • MMLU：测试模型在各个学科的知识掌握程度
  • HumanEval：测试模型写代码的能力
  • GSM8K：测试数学推理能力
  • 自定义 Benchmark：针对你的业务场景设计的测试集（比如”客服场景下 100 个常见问题的回答质量”）

Benchmark 的意义在于量化。它让你能客观地说”这个模型在这个任务上得分 85，那个模型得分 72”，而不是凭感觉说”好像这个更好”。

2. 生产环境监控

Benchmark 是在”考试环境”中测的，但真实世界更复杂。生产环境的 Harness 负责持续监控 AI 的实际表现：

  • 回复是否符合预期的格式？（比如要求输出 JSON，它真的输出合法 JSON 了吗？）
  • 回复是否包含有害内容？（涉黄、涉暴、歧视性言论）
  • 工具调用是否正确？（有没有该调用的时候没调用，或者调用了错误的参数？）
  • 延迟是否在可接受范围内？

3. 对齐与纠偏

当 Harness 发现问题后，需要有一套机制来”纠偏”。这可能包括：

  • 优化 Prompt，让模型更稳定地输出正确格式
  • 调整工具调用的参数或逻辑
  • 在关键路径上增加规则校验（比如 AI 生成的 SQL 必须先经过语法检查）
  • 收集用户反馈，形成”问题案例库”，持续迭代

Harness 的存在提醒我们一个重要的事实：AI 系统不是”部署了就完了”，它需要像传统软件一样，持续测试、监控、迭代。而且因为 AI 的非确定性，这套质检流程甚至比传统软件更复杂、更重要。

常见误区：新手最容易踩的坑

在结束之前，我想澄清几个常见的误解：

❌ 误区一：LLM 越大数据量越好，所以我把所有资料都塞进 Prompt
✅ 正确做法：Context 有长度限制，而且”信息过载”会让模型抓不住重点。只放最相关的信息，必要时用 RAG（检索增强生成）先筛选再输入。

❌ 误区二：Agent 就是更聪明的 AI，有了 Agent 就不需要人了
✅ 正确做法：Agent 的”自主”是有限的。它适合有明确步骤、可验证结果的任务，但在需要创造力、伦理判断、复杂人际沟通的场合，人类依然是不可替代的。Agent 的最佳定位是”辅助”，不是”替代”。

❌ 误区三：Prompt 工程就是写更好的”咒语”，骗 AI 说出我想要的答案
✅ 正确做法：Prompt 工程的本质是清晰的沟通。你不是在”骗”AI，你是在帮它缩小”猜词”的范围，让它更准确地激活相关知识区域。好的 Prompt 是精确的、结构化的、可复用的。

❌ 误区四：MCP 是某个公司的专有技术，用了会被绑定
✅ 正确做法：MCP 是一个开放协议，由 Anthropic 发起但任何人都可以实现。它类似于 HTTP 协议——不是某个公司的产品，而是整个行业的公共基础设施。

总结：一张图看懂 AI 生态

让我们回到开头的那张”地图”，现在你应该能理解每一层的含义了：

Harness（测试框架）         ← 确保一切靠谱运行
      ↑
Agent + Skills（智能体+技能） ← 自主规划、执行任务
      ↑
Tool + MCP（工具+协议）      ← 连接外部世界，标准化接入
      ↑
Prompt + Context（提示+上下文）← 控制模型输出
      ↑
LLM + Token（模型+信息原子）  ← 提供核心推理能力

这套体系的核心逻辑是：

  1. LLM 提供了原始的”猜词”能力，是整栋大厦的地基。
  2. Token 和 Context 定义了信息处理的物理边界——AI 能”看”多远、以什么单位处理。
  3. Prompt 是我们与 AI 交互的遥控器，决定了 AI 以什么状态、什么风格输出。
  4. Tool 弥补了 AI 的先天缺陷，让它能查实时数据、做精确计算、操作外部系统。
  5. MCP 让这一切连接变得标准化，不再是一家一个标准。
  6. Agent 把所有组件编织进一个自主循环，让 AI 从”回答问题”进化到”完成任务”。
  7. Skills 让 Agent 专业化，针对不同场景快速定制能力。
  8. Harness 则是整套系统的安全带和质检部，确保它在生产环境中稳定、可靠、可控。

未来的 AI 应用，竞争的核心不在”谁的模型更大”，而在谁能用这些组件搭出最稳定、最高效、最可靠的智能系统。理解这九个概念，你就已经站在了正确的起跑线上。

下一步可以学什么？

  • 如果你想动手实践：学习 Prompt Engineering 和 Function Calling，用 OpenAI/Claude API 做一个能查天气的聊天机器人
  • 如果你想深入系统架构：了解 RAG（检索增强生成） 和 向量数据库，让 AI 能基于你的私有知识库回答问题
  • 如果你想构建复杂应用：研究 LangChain / LlamaIndex 等 Agent 框架，学习如何编排多步骤任务
