OpenSpec 与 Specify 入门：规范驱动开发让 AI 不再幻觉

AI Spec-Driven Development Claude Code Cursor 工程实践
发布日期:   2026-04-21

一句话摘要：本文介绍规范驱动开发（Spec-Driven Development）的理念，对比 OpenSpec 与 Specify（GitHub Spec Kit）两款工具的差异，并给出各自的安装和使用方式，帮你让 AI 编程助手从”拍脑袋写代码”变成”按规范施工”。

背景与问题

用 Claude Code、Cursor 或 GitHub Copilot 写代码久了，你一定会遇到一个经典场景：

你告诉 AI “给我加一个用户认证功能”，它唰唰唰生成了一堆代码。看起来没问题，但跑起来发现：

  • 密码用了明文存储，而不是 bcrypt
  • JWT Token 没有设置过期时间
  • 登录接口的返回结构跟现有 API 完全不统一

你让它改，它改了 A 又破坏了 B。三个回合下来，你的上下文窗口爆了，AI 开始”失忆”，甚至把刚才修好的地方又改回去了。

这就是 AI 幻觉 在工程层面的表现：AI 没有”全局观”，它只根据你当前的提示和有限的上下文做局部最优解，而不是按照整个项目的规范来执行。

规范驱动开发（Spec-Driven Development） 就是来解决这个问题的。核心思路很简单：在让 AI 写代码之前，先让它（或你自己）写一份规范文档（Spec），把要做什么、怎么做、做到什么标准说清楚。AI 后续的所有代码生成，都严格基于这份规范，而不是凭空发挥。

核心原理

Spec 驱动开发借鉴了建筑行业的施工图纸逻辑：

  • 没有图纸，工人（AI）只能凭经验搭积木，搭完发现承重墙位置不对，拆了重来。
  • 有了图纸，工人按图施工，监理（你）按图验收，每一道工序都有据可查。

在技术实现上，Spec 驱动开发引入了一个关键机制：Delta Spec（差异规范）。每一次功能变更都被显式标注为 ADDED（新增）、MODIFIED（修改）或 REMOVED（删除），而不是直接覆盖原有代码。这迫使 AI（和人类）在动手之前，先想清楚变更对现有系统的影响。

工具简介：OpenSpec 与 Specify

目前生态中最成熟的两款 Spec 驱动开发工具分别是 OpenSpec 和 Specify（GitHub Spec Kit 的命令行工具）。

OpenSpec

OpenSpec 由 Fission AI 开源，定位是现有项目（Brownfield）的增量变更管理。它采用轻量级的三阶段工作流：提案（Proposal）→ 应用（Apply）→ 归档（Archive）。

安装方式（需要 Node.js 18+）：

npm install -g @fission-ai/openspec

初始化项目：

openspec init --tools claude

执行后会在项目根目录生成 openspec/ 文件夹：

openspec/
├── specs/          # 当前系统的权威规范（Source of Truth）
├── changes/        # 每个功能的变更提案
│   └── {feature}/
│       ├── proposal.md   # 变更提案
│       ├── tasks.md      # 任务清单
│       └── delta/        # 差异规范（ADDED/MODIFIED/REMOVED）
└── project.md      # 项目概述

在 Claude Code 中使用 OpenSpec 的典型流程：

# 1. 创建新功能提案
/opsx:new user-auth

# 2. AI 根据 proposal.md 编写代码
/opsx:apply

# 3. 验证实现是否符合规范
/opsx:verify

# 4. 归档，将 delta 合并回主规范
/opsx:archive

Specify（GitHub Spec Kit）

Specify 是 GitHub 官方推出的 Spec Kit 的命令行入口，定位是从零开始的新项目（Greenfield）。它采用更严格的五阶段工作流：原则（Principles）→ 规范（Specification）→ 计划（Plan）→ 任务（Tasks）→ 实现（Implement）。

安装方式（需要 uv）：

uv tool install specify-cli --from git+https://github.com/github/spec-kit.git

初始化项目：

specify init my-project --ai claude

在 Claude Code 中使用 Specify 的典型流程：

# 1. 定义系统规范
/speckit.specify

# 2. 生成实施计划
/speckit.plan

# 3. 拆解任务清单
/speckit.tasks

# 4. 按任务实现代码
/speckit.implement

核心区别对比

维度 OpenSpec Specify（Spec Kit）
出品方 Fission AI GitHub
适用场景 现有项目迭代（Brownfield） 从零开发新项目（Greenfield）
安装方式 npm install -g @fission-ai/openspec uv tool install specify-cli
规范体量 ~250 行（轻量） ~800 行（全面）
流程阶段 3 阶段：提案 → 应用 → 归档 5 阶段：原则 → 规范 → 计划 → 任务 → 实现
变更追踪 Delta Spec + 归档历史 无内置变更隔离机制
自由度 非线性，可随时回改提案 线性流程，需按步骤执行
Token 消耗 较低 较高

简单记忆法：

  • 已有代码库，做功能迭代或 Bug 修复 → 选 OpenSpec。它的 Delta Spec 机制天然适合”在现有大厦里装修”。
  • 从零开始新项目，需要完整架构设计 → 选 Specify。它的五阶段流程能帮你把地基打牢。

实战：用 OpenSpec 给现有项目加一个缓存层

假设你有一个 FastAPI 项目，想用 OpenSpec 规范驱动的方式给所有列表查询接口加上 Redis 缓存。

步骤 1：初始化 OpenSpec

# 在项目根目录执行
openspec init --tools claude

步骤 2：创建变更提案

/opsx:new add-redis-cache

Claude Code 会在 openspec/changes/add-redis-cache/ 下生成 proposal.md，内容类似：

# Proposal: Add Redis Cache for List Endpoints

## 目标
为所有列表查询接口添加 Redis 缓存，减少数据库压力。

## Delta Spec

### ADDED
- `app/core/cache.py`：Redis 连接管理与缓存装饰器
- `app/dependencies/cache.py`：FastAPI Depends 缓存注入

### MODIFIED
- `app/api/items.py`：`list_items()` 增加缓存读取逻辑
- `app/api/users.py`：`list_users()` 增加缓存读取逻辑

### REMOVED
- 无

## 验收标准
- 首次请求命中数据库，二次请求命中 Redis（< 5ms）
- 缓存过期时间统一为 300 秒
- 所有缓存 key 带前缀 `fastapi:list:`

步骤 3：应用规范

/opsx:apply

Claude Code 读取 proposal.md 和 Delta Spec，只修改 MODIFIED 中指定的文件，只新增 ADDED 中指定的文件。不会碰其他无关代码。

步骤 4：验证与归档

/opsx:verify
/opsck:archive

验证通过后，OpenSpec 会把这次的 Delta Spec 合并到 openspec/specs/ 下的主规范中，变更历史永久保留。

实战：用 Specify 从零搭建一个任务队列服务

假设你要从零开始一个 Celery + Redis 的任务队列服务。

步骤 1：初始化项目

specify init task-service --ai claude
cd task-service

步骤 2：定义规范

/speckit.specify

你需要告诉 AI 系统的核心约束，例如：

- 使用 FastAPI 作为 Web 框架
- 使用 Celery + Redis 作为任务队列
- 所有任务必须支持重试，最大重试 3 次
- 任务状态必须持久化到 PostgreSQL

Specify 会生成一份完整的系统规范文档（specs/specification.md），包含数据模型、API 接口、任务定义等。

步骤 3：生成计划与任务

/speckit.plan
/speckit.tasks

AI 会把规范拆解成可执行的任务清单（tasks.md），例如：

- [ ] 搭建 FastAPI 项目骨架
- [ ] 配置 Celery 与 Redis 连接
- [ ] 定义任务模型与数据库表
- [ ] 实现任务提交接口
- [ ] 实现任务状态查询接口
- [ ] 编写单元测试

步骤 4：按任务实现

/speckit.implement

AI 按照 tasks.md 逐项实现，每完成一项打勾。你可以随时暂停、回退或调整规范。

踩坑记录

❌ 错误做法：把 OpenSpec 和 Specify 混着用

有人觉得 OpenSpec 的 Delta 机制好，Specify 的规范全面，于是两个都装。结果同一个项目里既有 openspec/ 又有 specs/ 目录，AI 完全懵掉，不知道应该遵循哪份规范。

✅ 正确做法：一个项目只选一个工具。Brownfield 选 OpenSpec，Greenfield 选 Specify，不要混用。

❌ 错误做法：规范写得太细，成了”伪代码文学”

有人写 Spec 时把每一行代码都描述出来，比如”第 1 行 import redis，第 2 行定义函数…”，结果规范本身几百行，AI 的上下文窗口被吃光，代码生成质量反而下降。

✅ 正确做法：规范只描述做什么和验收标准，不描述怎么做。把实现细节留给 AI。OpenSpec 推荐的 ~250 行规范体量是经过验证的甜点。

❌ 错误做法：归档后删掉 changes/ 目录

OpenSpec 的 changes/ 目录归档后会被移动到 changes/archive/，这是刻意设计的。有人为了”整洁”直接删掉，结果变更历史丢失，后续排查问题无据可查。

✅ 正确做法：保留归档目录，它是项目的变更日志。如果确实需要清理，只删已合并到主规范且超过半年的旧归档。

总结

规范驱动开发不是给 AI 上枷锁，而是给它一张施工图纸。OpenSpec 和 Specify 是两张不同场景的图纸：前者轻量灵活，适合在现有项目里做增量变更；后者全面严谨，适合从零搭建新项目。

下一步可以学什么：

  • 把 OpenSpec 或 Specify 集成到你的 CI/CD 流程中，实现”规范变更必须审批，代码实现必须匹配规范”
  • 探索 Maestro 等工具，将 Spec 驱动开发与多 Agent 协作结合
  • 在团队内推行 Spec Review（规范评审），把代码评审前移一层
