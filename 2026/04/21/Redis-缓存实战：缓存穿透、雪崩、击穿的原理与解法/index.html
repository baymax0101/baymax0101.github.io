Redis 缓存三大问题：穿透、雪崩、击穿的原理与 Python 解法

FastAPI Python Redis 后端 缓存
FastAPI 实战
发布日期:   2026-04-21

本文是《FastAPI 实战》系列第 5 篇。上篇我们讲了 JWT 双 Token 机制实战：Access Token + Refresh Token 完整实现，完成了用户认证层的搭建。本文解决一个更实际的问题：当流量上来后，Redis 缓存怎么就成了系统的最大隐患？

背景与问题

你的接口响应越来越慢，数据库 CPU 飙到 90%，你加了 Redis 缓存，QPS 瞬间提升了 10 倍。但上线第二天，数据库又挂了。

排查日志发现，大量请求直接打到了数据库——缓存没拦住。这不是 Redis 的锅，而是你踩了缓存三大经典陷阱：穿透、雪崩、击穿。这三兄弟名字长得像，原理和解法完全不同，面试里也是高频考点，但很多人背完概念还是写不出能扛住高并发的代码。

本文先讲清楚三个问题的本质区别，再给出一套可以直接用在 FastAPI 项目里的 Python 实现。

核心原理

用一个场景串讲：你的博客系统用 Redis 缓存文章详情，key 是 article:{id}，过期时间 1 小时。

问题 触发场景 本质
缓存穿透 黑客用 id=-99999 持续请求 查询一个不存在的数据，缓存不会命中，请求直达数据库
缓存雪崩 凌晨 4 点，所有文章的缓存同时过期 大量 key 在同一时间失效，请求洪峰瞬间压垮数据库
缓存击穿 某篇爆款文章缓存过期，瞬间几万请求涌入 单个热点 key 失效瞬间，并发请求同时打到数据库

一句话区分：穿透是查不存在的东西，雪崩是大家一起过期，击穿是热点 key 过期被围殴。

环境准备

pip install fastapi uvicorn redis[hiredis]
  • redis：官方 async Redis 客户端（5.0+ 版本原生支持 async/await）
  • hiredis：C 语言解析器，提升序列化性能

实战代码

1. Redis 连接封装

先把 Redis 连接做成可复用的依赖，方便在 FastAPI 中注入。

# app/cache.py
import os
import json
from typing import Any, Optional

import redis.asyncio as redis

REDIS_URL = os.getenv("REDIS_URL", "redis://localhost:6379")


class RedisCache:
    def __init__(self, url: str = REDIS_URL):
        self._client = redis.from_url(url, decode_responses=True)

    async def get(self, key: str) -> Optional[str]:
        return await self._client.get(key)

    async def set(
        self,
        key: str,
        value: str,
        expire: int = 3600,
    ) -> None:
        await self._client.setex(key, expire, value)

    async def delete(self, key: str) -> None:
        await self._client.delete(key)

    async def setnx(self, key: str, value: str, expire: int) -> bool:
        """SET if Not eXists — 分布式锁的核心命令。"""
        return await self._client.set(key, value, nx=True, ex=expire)

    async def close(self) -> None:
        await self._client.close()


cache = RedisCache()

FastAPI 的依赖注入配合 async 非常自然，路由里直接 Depends(get_cache) 即可拿到实例。

2. 缓存穿透：布隆过滤器 + 空值缓存

核心思路：既然攻击者查询的是不存在的数据，那我们提前告诉他”这东西不存在”，别去问数据库了。

方案 A：空值缓存（简单有效）

# app/cache.py

async def get_article_with_null_cache(
    article_id: int,
    db: AsyncSession,
) -> Optional[dict]:
    cache_key = f"article:{article_id}"

    # 1. 查缓存
    cached = await cache.get(cache_key)
    if cached is not None:
        # 空值缓存用特殊标记 "" 表示"数据库里不存在"
        if cached == "__NULL__":
            return None
        return json.loads(cached)

    # 2. 查数据库
    result = await db.execute(
        select(Article).where(Article.id == article_id)
    )
    article = result.scalar_one_or_none()

    # 3. 写缓存 —— 数据库里没有也要缓存，防穿透
    if article is None:
        await cache.set(cache_key, "__NULL__", expire=60)  # 空值缓存时间短一点
        return None

    data = {
        "id": article.id,
        "title": article.title,
        "content": article.content,
    }
    await cache.set(cache_key, json.dumps(data), expire=3600)
    return data

关键点：数据库返回 None 时，缓存里写 "__NULL__"，下次同样请求直接从缓存返回 None，不会打到数据库。空值缓存的过期时间建议设短（30~60 秒），防止数据实际已写入但缓存还是空值。

方案 B：布隆过滤器（更省内存）

如果你的业务场景是”大量 id 中只存在极少数”（比如用 UUID 做短链），空值缓存会占用大量 Redis 内存。这时候用布隆过滤器更优雅——它是一种概率型数据结构，能 100% 判断”一定不存在”，可能误判”存在”。

# app/cache.py
import redis.asyncio as redis

# 需要 Redis 4.0+ 并启用 RedisBloom 模块
async def init_bloom_filter(client: redis.Redis, capacity: int = 1_000_000):
    """初始化布隆过滤器，用于判断文章 ID 是否可能存在。"""
    try:
        await client.execute_command(
            "BF.RESERVE", "articles:bf", 0.001, capacity
        )
    except redis.ResponseError:
        pass  # 已存在则忽略


async def add_to_bloom(client: redis.Redis, article_id: int):
    await client.execute_command("BF.ADD", "articles:bf", article_id)


async def exists_in_bloom(client: redis.Redis, article_id: int) -> bool:
    """返回 0 表示一定不存在，返回 1 表示可能存在。"""
    result = await client.execute_command(
        "BF.EXISTS", "articles:bf", article_id
    )
    return bool(result)

使用布隆过滤器时，查询流程变为：先查布隆过滤器 → 如果不存在直接返回 404 → 如果可能存在，再查缓存/数据库。写数据时同步把 id 加入布隆过滤器。

注意：Redis 原生不支持布隆过滤器，需要安装 RedisBloom 模块，或用 pybloom-live 在应用层实现。

3. 缓存雪崩：随机过期时间 + 多级缓存

核心思路：别让所有 key 在同一个时间点过期。

方案 A：过期时间加随机偏移

# app/cache.py
import random

BASE_EXPIRE = 3600  # 基础过期时间 1 小时


async def set_with_jitter(key: str, value: str, base_expire: int = BASE_EXPIRE):
    """给过期时间加一个随机偏移，避免同时失效。"""
    jitter = random.randint(0, 300)  # 0~5 分钟随机偏移
    await cache.set(key, value, expire=base_expire + jitter)

就是这么简单，但效果极好。10000 个 key 的过期时间被均匀打散在 1 小时到 1 小时 5 分钟之间，数据库不会在同一时刻承受洪峰。

方案 B：热点数据永不过期 + 异步刷新

对于核心配置、热门文章等真正重要的数据，可以干脆不设过期时间，用后台任务异步更新：

# app/cache.py
import asyncio


async def refresh_hot_articles():
    """后台协程：每 10 分钟刷新热门文章缓存。"""
    while True:
        # 从数据库读取热门文章列表
        hot_ids = [1, 3, 7, 42]  # 实际从统计表获取
        for aid in hot_ids:
            # 重新加载并更新缓存，不修改过期时间（永不过期）
            article = await fetch_article_from_db(aid)
            if article:
                await cache.set(f"article:{aid}", json.dumps(article))
        await asyncio.sleep(600)

在 FastAPI 中启动后台任务：

# app/main.py
@app.on_event("startup")
async def startup_event():
    asyncio.create_task(refresh_hot_articles())

4. 缓存击穿：分布式锁 + 逻辑过期

核心思路：热点 key 过期时，只允许一个请求去重建缓存，其他请求要么等、要么用旧数据。

方案 A：分布式互斥锁（SET NX EX）

# app/cache.py
import uuid

LOCK_PREFIX = "lock:"
LOCK_TIMEOUT = 10  # 锁最长持有 10 秒，防止死锁


async def get_article_with_lock(
    article_id: int,
    db: AsyncSession,
) -> Optional[dict]:
    cache_key = f"article:{article_id}"
    lock_key = f"{LOCK_PREFIX}{cache_key}"
    request_id = str(uuid.uuid4())

    # 1. 查缓存
    cached = await cache.get(cache_key)
    if cached:
        return json.loads(cached)

    # 2. 尝试获取分布式锁
    acquired = await cache.setnx(lock_key, request_id, LOCK_TIMEOUT)

    if acquired:
        try:
            # 只有拿到锁的线程才能查数据库
            result = await db.execute(
                select(Article).where(Article.id == article_id)
            )
            article = result.scalar_one_or_none()

            if article:
                data = {
                    "id": article.id,
                    "title": article.title,
                    "content": article.content,
                }
                await set_with_jitter(cache_key, json.dumps(data))
                return data
            else:
                await cache.set(cache_key, "__NULL__", expire=60)
                return None
        finally:
            # 释放锁（用 Lua 脚本保证原子性）
            await cache._client.eval(
                "if redis.call('get', KEYS[1]) == ARGV[1] then "
                "return redis.call('del', KEYS[1]) else return 0 end",
                1, lock_key, request_id,
            )
    else:
        # 3. 没拿到锁，短暂等待后重试（或返回降级数据）
        await asyncio.sleep(0.1)
        cached = await cache.get(cache_key)
        return json.loads(cached) if cached else None

这段代码的逻辑：缓存失效 → 多个并发请求同时到达 → 只有一个请求拿到锁去重建缓存 → 其他请求短暂等待后读新缓存。锁用 SET key value NX EX 实现，释放时用 Lua 脚本保证”判断 value 相等再删除”的原子性，防止误删别人的锁。

方案 B：逻辑过期（永不阻塞）

分布式锁会让没拿到锁的请求等待或失败。对于要求”绝对低延迟”的场景，可以用逻辑过期：缓存里存的数据自带过期时间戳，而不是依赖 Redis 的 TTL。

# app/cache.py
from datetime import datetime, timedelta, timezone


async def get_article_logical_expire(
    article_id: int,
    db: AsyncSession,
) -> Optional[dict]:
    cache_key = f"article:{article_id}"
    cached = await cache.get(cache_key)

    now = datetime.now(timezone.utc)

    if cached:
        data = json.loads(cached)
        # 检查逻辑过期时间
        expire_at = datetime.fromisoformat(data["_expire_at"])

        if expire_at > now:
            # 未过期，直接返回
            return data["payload"]

        # 已过期，但数据还在 —— 返回旧数据，同时异步重建
        asyncio.create_task(rebuild_cache(article_id, db))
        return data["payload"]

    # 缓存完全不存在，走正常重建流程
    return await rebuild_cache(article_id, db)


async def rebuild_cache(article_id: int, db: AsyncSession) -> Optional[dict]:
    """查数据库并写入带逻辑过期时间的缓存。"""
    result = await db.execute(
        select(Article).where(Article.id == article_id)
    )
    article = result.scalar_one_or_none()

    if not article:
        return None

    payload = {
        "id": article.id,
        "title": article.title,
        "content": article.content,
    }

    # 逻辑过期时间 = 物理 TTL 的 80%，留 20% 窗口做后台刷新
    expire_at = datetime.now(timezone.utc) + timedelta(seconds=2880)

    wrapped = {
        "payload": payload,
        "_expire_at": expire_at.isoformat(),
    }

    # 物理 TTL 设长一点（比如 30 天），靠逻辑过期控制
    await cache.set(f"article:{article_id}", json.dumps(wrapped), expire=2592000)
    return payload

逻辑过期的核心优势：任何请求都不会因为缓存重建而阻塞。即使后台重建失败，用户仍然能看到旧数据，而不是 500 错误。代价是代码复杂度稍高，且旧数据可能有短暂延迟。

5. 统一封装：一个函数解决三大问题

实际项目中，可以把上述策略组合成一个通用的缓存装饰器/工具函数：

# app/cache.py
from functools import wraps
from typing import Callable, TypeVar

T = TypeVar("T")


async def cached_get(
    key: str,
    fetcher: Callable[[], T],
    expire: int = 3600,
    null_expire: int = 60,
    use_lock: bool = True,
) -> T:
    """
    通用缓存读取，自动处理穿透、击穿、雪崩。

    - 穿透：空值缓存（__NULL__）
    - 雪崩：随机过期偏移
    - 击穿：分布式锁（可选）
    """
    # 1. 查缓存
    cached = await cache.get(key)
    if cached == "__NULL__":
        return None
    if cached:
        return json.loads(cached)

    # 2. 防击穿：分布式锁
    if use_lock:
        lock_key = f"{LOCK_PREFIX}{key}"
        request_id = str(uuid.uuid4())
        acquired = await cache.setnx(lock_key, request_id, LOCK_TIMEOUT)

        if not acquired:
            # 等 100ms 再读一次缓存
            await asyncio.sleep(0.1)
            cached = await cache.get(key)
            return json.loads(cached) if cached and cached != "__NULL__" else None

    try:
        # 3. 查数据库
        data = await fetcher()

        # 4. 写缓存（带随机偏移防雪崩）
        if data is None:
            await cache.set(key, "__NULL__", expire=null_expire)
        else:
            await set_with_jitter(key, json.dumps(data), base_expire=expire)

        return data
    finally:
        if use_lock:
            await cache._client.eval(
                "if redis.call('get', KEYS[1]) == ARGV[1] then "
                "return redis.call('del', KEYS[1]) else return 0 end",
                1, lock_key, request_id,
            )

使用方式：

# app/api/articles.py
@router.get("/{article_id}")
async def get_article(
    article_id: int,
    db: AsyncSession = Depends(get_db),
):
    async def fetch():
        result = await db.execute(
            select(Article).where(Article.id == article_id)
        )
        article = result.scalar_one_or_none()
        if article:
            return {"id": article.id, "title": article.title}
        return None

    data = await cached_get(
        key=f"article:{article_id}",
        fetcher=fetch,
        expire=3600,
        use_lock=True,
    )

    if data is None:
        raise HTTPException(status_code=404, detail="文章不存在")
    return data

踩坑记录

❌ 用 TIME 命令做分布式锁的过期判断

有些教程让你用 EXPIRE 配合 TIME 命令判断锁是否超时，这在分布式系统里是错的——各节点时钟可能不同步。

# 错误做法：依赖系统时钟做超时判断
lock_time = await cache._client.time()  # 获取 Redis 服务器时间
# 然后自己算差值判断是否超时...  节点间时钟漂移会导致误判

✅ 正确做法：直接用 SET key value NX EX 10，让 Redis 原子性地设置键和过期时间，不需要客户端判断超时。

❌ 缓存重建抛出异常，导致锁永远不被释放

# 错误做法：没加 try/finally
acquired = await cache.setnx(lock_key, request_id, 10)
if acquired:
    data = await fetch_from_db()   # 如果这里抛异常...
    await cache.set(key, data)     # ...锁永远不会被释放
    await cache.delete(lock_key)   # 这一行执行不到

✅ 正确做法：锁的释放必须放在 finally 块里。更好的方案是给锁加一个较短的自动过期时间（如 10 秒），即使代码异常，锁也会自动释放，只是可能多等几秒。

❌ 空值缓存时间设太长，导致”刚写入的数据读不到”

# 错误做法：空值缓存和正常数据一样长
await cache.set(key, "__NULL__", expire=3600)  # 1 小时内即使数据库有了数据，缓存还是空值

✅ 正确做法：空值缓存时间设短（30~60 秒），业务能容忍的”写入后延迟可见”窗口内即可。如果要求强一致，写入数据库后主动 DELETE 对应的空值缓存 key。

❌ 缓存和数据库更新顺序不当

先删缓存再写数据库，中间并发读请求会把旧数据重新写回缓存：

# 错误顺序（竞态条件）
await cache.delete("article:1")     # 缓存删除
# ← 另一个线程在这里读数据库，拿到旧数据，写入缓存
await db.execute(update_article)    # 数据库更新

✅ 正确做法：先更新数据库，再删除缓存（Cache-Aside 模式）。或者使用延迟双删：先删缓存 → 更新数据库 → 等 500ms → 再删一次缓存。

总结

缓存三大问题的核心解法可以归纳为一张表：

问题 核心思路 推荐方案
缓存穿透 拦截不存在的数据查询 空值缓存（简单）/ 布隆过滤器（省内存）
缓存雪崩 打散过期时间点 随机过期偏移 + 热点数据永不过期
缓存击穿 限制并发重建 分布式锁（强一致）/ 逻辑过期（高可用）

实际项目中，空值缓存 + 随机过期 + 分布式锁 的组合已经能应对绝大多数场景。逻辑过期适合对延迟极度敏感的业务，布隆过滤器适合数据量极大且查询模式固定的场景。
