JWT 双 Token 机制实战：Access Token + Refresh Token 完整实现

FastAPI Python JWT 后端 认证
FastAPI 实战
发布日期:   2026-04-21

本文是《FastAPI 实战》系列第 4 篇。上篇我们讲了 SQLAlchemy 2.0 + Alembic：数据库建模与版本迁移，完成了数据库层的搭建。本文解决一个更实际的问题：用户登录后，如何安全地维持会话状态，并让 Token 自动续期？

背景与问题

你的 API 已经能读写数据库了，但用户一刷新页面，就需要重新登录——这体验太差了。

传统的 Session-Cookie 方案在前后端分离的场景下并不优雅：前端可能是 Vue/React 移动端 App，甚至第三方服务调用你的 API。跨域、跨端、无状态的 HTTP 请求，让”维持登录状态”这件事变得复杂。

JWT（JSON Web Token）就是为了解决这个问题而生。但很多教程只讲了”签发一个 Token，每次请求带上去”，却忽略了几个关键问题：

  • Token 过期了怎么办？让用户重新登录？
  • 如果 Token 被截获，攻击者能一直冒充用户吗？
  • 用户点击”退出登录”，如何让已经发出的 Token 失效？

这些问题，用双 Token 机制（Access Token + Refresh Token）都能解决。

核心原理

JWT 本质上是一个带签名的 JSON 字符串，由三部分组成，用 . 分隔：

xxxxx.yyyyy.zzzzz
↑      ↑      ↑
Header Payload Signature
  • Header：声明类型和签名算法（通常是 HS256）
  • Payload：携带用户 ID、角色、过期时间等声明
  • Signature：用密钥对前两部分做签名，防止篡改

双 Token 机制的核心思想是”分级管理”：

  • Access Token：有效期短（15 分钟），负责日常接口鉴权，频繁暴露在网络中
  • Refresh Token：有效期长（7 天），只用于换取新的 Access Token，不直接调用业务接口

这样做的好处是：即使 Access Token 泄露，攻击者最多只能冒充用户 15 分钟。而 Refresh Token 只在”换票”时使用，暴露面极小。

环境准备

Python 版本建议 3.10+，核心依赖如下：

pip install fastapi uvicorn python-jose[cryptography] passlib[bcrypt] python-multipart
  • python-jose：JWT 的生成与解析
  • passlib[bcrypt]：密码哈希
  • python-multipart：FastAPI 解析表单登录数据

项目沿用前三篇的目录结构：

app/
├── __init__.py
├── main.py
├── models.py          # SQLAlchemy 模型
├── schemas.py         # Pydantic 校验模型
├── database.py        # 数据库连接
├── auth.py            # 认证核心逻辑 ← 本文重点
└── api/
    ├── __init__.py
    └── auth.py        # 登录/刷新接口

实战代码

1. 配置与安全参数

密钥和过期时间一定要走环境变量，别写死在代码里。

# app/config.py
import os
from datetime import timedelta

SECRET_KEY = os.getenv("SECRET_KEY", "change-me-in-production")
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 15
REFRESH_TOKEN_EXPIRE_DAYS = 7

生产环境可以用 openssl rand -hex 32 生成密钥，写入 .env：

SECRET_KEY=09d25e094faa6ca2556c818166b7a9563b93f7099f6f0f4caa6cf63b88e8d3e7

2. 密码哈希与校验

明文存密码是安全红线，passlib 帮你做了这件事。

# app/auth.py
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")


def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)


def get_password_hash(password: str) -> str:
    return pwd_context.hash(password)

注册时调用 get_password_hash(password) 入库，登录时用 verify_password() 比对。

3. Token 生成与解析

这是认证系统的核心。我们生成两种 Token，并封装一个”解析并校验 Token”的依赖函数。

# app/auth.py
from datetime import datetime, timedelta, timezone
from typing import Optional
from jose import JWTError, jwt
from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select

from app.config import SECRET_KEY, ALGORITHM, ACCESS_TOKEN_EXPIRE_MINUTES, REFRESH_TOKEN_EXPIRE_DAYS
from app.database import get_db
from app.models import User

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/api/v1/auth/login")


def create_token(subject: str, expires_delta: timedelta, token_type: str) -> str:
    now = datetime.now(timezone.utc)
    to_encode = {
        "sub": subject,
        "iat": now,
        "exp": now + expires_delta,
        "type": token_type,
    }
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)


def create_access_token(user_id: str) -> str:
    return create_token(
        user_id,
        timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES),
        "access",
    )


def create_refresh_token(user_id: str) -> str:
    return create_token(
        user_id,
        timedelta(days=REFRESH_TOKEN_EXPIRE_DAYS),
        "refresh",
    )

注意到 Payload 里加了 "type": "access" 和 "type": "refresh"，防止 Refresh Token 被当成 Access Token 使用。

接下来封装一个可复用的依赖函数，供需要登录态的接口直接 Depends(get_current_user)：

# app/auth.py

async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="无效的认证凭据",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: Optional[str] = payload.get("sub")
        token_type: Optional[str] = payload.get("type")

        if user_id is None or token_type != "access":
            raise credentials_exception
    except JWTError:
        raise credentials_exception

    result = await db.execute(select(User).where(User.id == user_id))
    user = result.scalar_one_or_none()

    if user is None:
        raise credentials_exception

    return user

这段代码的逻辑链：提取 Bearer Token → 解析 JWT → 校验类型必须是 access → 查数据库确认用户存在 → 返回 User 对象。任何一个环节失败，都统一返回 401。

4. 登录与 Token 刷新接口

# app/api/auth.py
from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select

from app.database import get_db
from app.models import User
from app.schemas import TokenResponse, LoginRequest
from app.auth import (
    verify_password,
    create_access_token,
    create_refresh_token,
    get_current_user,
)

router = APIRouter(prefix="/api/v1/auth", tags=["认证"])


@router.post("/login", response_model=TokenResponse)
async def login(
    payload: LoginRequest,
    db: AsyncSession = Depends(get_db),
):
    result = await db.execute(
        select(User).where(User.username == payload.username)
    )
    user = result.scalar_one_or_none()

    if not user or not verify_password(payload.password, user.hashed_password):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="用户名或密码错误",
        )

    return {
        "access_token": create_access_token(str(user.id)),
        "refresh_token": create_refresh_token(str(user.id)),
        "token_type": "bearer",
    }

登录接口的逻辑很直观：校验用户名密码 → 生成双 Token → 返回给前端。前端通常把 Access Token 存在内存（如 Vuex/Pinia），Refresh Token 放在 HttpOnly Cookie 中，防止 XSS 窃取。

再看刷新接口：

# app/api/auth.py
from jose import jwt, JWTError
from app.config import SECRET_KEY, ALGORITHM


@router.post("/refresh", response_model=TokenResponse)
async def refresh_access_token(
    refresh_token: str,
    db: AsyncSession = Depends(get_db),
):
    try:
        payload = jwt.decode(refresh_token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: str | None = payload.get("sub")
        token_type: str | None = payload.get("type")

        if user_id is None or token_type != "refresh":
            raise HTTPException(status_code=401, detail="无效的 Refresh Token")
    except JWTError:
        raise HTTPException(status_code=401, detail="无效的 Refresh Token")

    result = await db.execute(select(User).where(User.id == user_id))
    user = result.scalar_one_or_none()

    if user is None:
        raise HTTPException(status_code=401, detail="用户不存在")

    return {
        "access_token": create_access_token(user_id),
        "refresh_token": create_refresh_token(user_id),
        "token_type": "bearer",
    }

刷新接口同样校验 Token 类型必须是 refresh。成功后，一次性把两个 Token 都换掉（Refresh Token Rotation），这样即使旧的 Refresh Token 被截获，也只能用一次。

5. 保护业务接口

有了 get_current_user，保护接口就是一行代码的事：

# app/api/users.py
from fastapi import APIRouter, Depends
from app.auth import get_current_user
from app.models import User

router = APIRouter(prefix="/api/v1/users", tags=["用户"])


@router.get("/me")
async def read_current_user(current_user: User = Depends(get_current_user)):
    return {
        "id": current_user.id,
        "username": current_user.username,
        "email": current_user.email,
    }

Depends(get_current_user) 会自动提取请求头中的 Bearer Token，解析校验，并将 User 对象注入到 current_user 参数中。如果 Token 无效或过期，FastAPI 会直接返回 401，你的业务代码完全不用关心鉴权细节。

6. 注册接口（供完整测试）

# app/api/auth.py
from app.auth import get_password_hash
from app.schemas import UserCreate, UserResponse


@router.post("/register", response_model=UserResponse, status_code=201)
async def register(
    payload: UserCreate,
    db: AsyncSession = Depends(get_db),
):
    result = await db.execute(
        select(User).where(User.username == payload.username)
    )
    if result.scalar_one_or_none():
        raise HTTPException(status_code=400, detail="用户名已存在")

    user = User(
        username=payload.username,
        email=payload.email,
        hashed_password=get_password_hash(payload.password),
    )
    db.add(user)
    await db.commit()
    await db.refresh(user)
    return user

7. Pydantic 模型定义

# app/schemas.py
from pydantic import BaseModel, EmailStr


class LoginRequest(BaseModel):
    username: str
    password: str


class TokenResponse(BaseModel):
    access_token: str
    refresh_token: str
    token_type: str


class UserCreate(BaseModel):
    username: str
    email: EmailStr
    password: str


class UserResponse(BaseModel):
    id: int
    username: str
    email: str

    class Config:
        from_attributes = True

踩坑记录

❌ 在 Payload 里放敏感信息

JWT 的 Payload 只是 Base64 编码，不是加密。任何人都能解码看到里面的内容。

# 错误做法：把密码甚至手机号写进 Payload
token = jwt.encode(
    {"sub": user_id, "password": user.password, "phone": user.phone},
    SECRET_KEY,
    algorithm=ALGORITHM,
)

✅ 正确做法：Payload 只放最小必要的标识（如用户 ID、角色），敏感信息一律不放。

# 正确做法：只放用户 ID 和角色
{"sub": str(user.id), "role": user.role, "type": "access"}

❌ 不校验 Token 类型，导致 Refresh Token 越权

如果你的 get_current_user 不区分 Token 类型，攻击者拿到 Refresh Token 就能直接访问业务接口。

# 错误做法：不检查 type 字段
payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
user_id = payload.get("sub")  # 不管是 access 还是 refresh 都放行

✅ 正确做法：get_current_user 必须校验 type == "access"，刷新接口校验 type == "refresh"。

❌ Token 过期时间设置不合理

  • Access Token 设成 7 天：泄露后攻击者有充足时间作恶
  • Access Token 设成 30 秒：用户体验极差，频繁触发刷新

✅ 建议：Access Token 15 分钟、Refresh Token 7 天，配合 Refresh Token Rotation（刷新时同时换新的 Refresh Token），是一个业界普遍接受的平衡。

❌ 用户退出登录后 Token 仍然有效

JWT 一旦签发，在过期前都是”合法”的。用户点了退出，Token 并不会自动失效。

✅ 解决方案：维护一个 Token 黑名单（通常用 Redis）。退出时把该 Token 的 jti（JWT ID）或过期时间戳写入 Redis，有效期设为 Token 剩余存活时间。校验时先查黑名单：

# 退出登录时
async def logout(token: str = Depends(oauth2_scheme)):
    payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    exp = payload.get("exp")
    # 写入 Redis，TTL 设为 Token 剩余时间
    await redis.setex(f"blacklist:{token}", exp - int(time.time()), "1")

总结

双 Token 机制的核心价值在于分级暴露：Access Token 短命且频繁使用，Refresh Token 长命但极少暴露。再配合 Refresh Token Rotation 和 Redis 黑名单，你能得到一个生产环境可用的认证方案。
