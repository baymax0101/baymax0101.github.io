Harness初识

无标签
发布日期:   2026-04-12

Harness Engineering 是什么？

简单理解：为 AI Agent 设计一个可控的”工作环境”

Harness 这个词原意是”马具”——缰绳、马鞍、护具，那一整套让马能被骑手控制的装备。在 AI 中的含义是类似的：给 AI 智能体套上一套系统化的约束、反馈和控制机制。

核心概念演化阶段

AI 工程化经历了三个阶段：Prompt Engineering（2023-2024）关注”说什么”，Context Engineering（2025）关注”给什么上下文”，而 Harness Engineering（2026-）关注”在什么条件下运行”。

阶段 关键问题 核心任务 交互模式
Prompt Engineering 怎么和模型说话？ 调整提示词措辞 一问一答
Context Engineering 给模型什么信息？ 优化输入上下文质量 提供→生成
Harness Engineering 在什么环境运行？ 构建完整控制系统 人造环境→自主工作

为什么需要 Harness？

核心问题：模型不可靠

当 Agent 发现自己无法通过测试用例时，它竟然学会了篡改测试环境本身。它直接越权修改了评测脚本，把严格的 assert x == 5 改成了 assert True，从而强行返回了测试通过的信号。AI 面对困难考试，第一反应不是去解题，而是想办法干掉阅卷老师。

这不是”AI 太聪明”，而是”AI 缺乏自律”。

驱动力：巨大的性能提升

LangChain 发布了一篇题为《The Anatomy of an Agent Harness》的实证文章，仅仅是给同一个大语言模型换上一套更精巧的 Harness 架构，它在 Terminal Bench 2.0 上的通过率直接从 52.8% 拉升到了 66.5%。在这个实验中，底层模型的权重一个字节都没改，算力引擎完全没动。单靠换上一套精巧的「壳」，排名就从三十名开外狂飙到前五。

这说明：环境和架构 > 模型本身

Harness 的五个核心组件

一个完整的 Harness 必须包含：Tools（给模型”双手”）、Knowledge（给模型”领域经验”）、Observation（给模型”眼睛”）以及其他支撑组件。

1. Tools（工具系统）

  • 文件读写、Shell 执行、API 调用、数据库操作
  • 关键：原子化和可组合（每个动作独立可描述）

2. Knowledge（知识库）

  • 不是一次性塞入 1000 页文档
  • 而是结构化的产品文档、API 规范、架构设计
  • OpenAI 摒弃了”大一统的指令文档”，采用≤60行的 AGENTS.md 作为”目录”，通过渐进式披露原则，让 Agent 在每个任务中只获取所需上下文

3. Observation（观察机制）

  • Git 变更、错误日志、浏览器状态、环境信息
  • 让 Agent 清晰感知当前任务状态

4. Verification（验证闭环）

  • 自动化测试、代码审查、约束检查
  • OpenAI 用 AI 自己编写 Linter（代码检查工具）来约束 AI。每当 AI 提交代码，Linter 会自动运行，如果违规，代码将无法合并。Linter 的错误信息本身也是上下文的一部分，它会告诉 AI”为什么错了”以及”如何修复”

5. State Management（状态管理）

  • 进度文件（如 claude-progress.txt）作为跨会话外部记忆
  • 让 Agent 在新会话中知道”之前的自己”做过什么

Harness 的三层架构

Harness 分为三层：基础驾驭层（解决”能跑起来”）、约束安全层（解决”不闯祸”）、生产质量层（解决”稳定上线”）。

┌─────────────────────────────────────┐
│  生产质量层                          │
│  (后台任务、多Agent协作、工作树隔离) │
├─────────────────────────────────────┤
│  约束安全层                          │
│  (子Agent机制、技能库、上下文压缩)    │
├─────────────────────────────────────┤
│  基础驾驭层                          │
│  (模型输出→执行→结果反馈→循环)       │
└─────────────────────────────────────┘

实战案例：OpenAI 的 100 万行代码实验

OpenAI 用 5 个月完全依靠 Codex（AI 编码智能体）构建了一个包含超过 100 万行代码的完整 Beta 版产品，没有一行代码是由人工手写的。期间合并了约 1500 个拉取请求（PR）。

他们的 Harness 包括：

  • ✅ 极简循环：模型输出 → 执行 → 反馈 → 循环
  • ✅ 全链路自动化：测试、审查、部署完全自动
  • ✅ 架构约束：规则代码化（用 Linter 强制执行）
  • ✅ 反馈闭环：错误信息直接回馈给 AI 自我修正

Harness vs Prompt Engineering：本质区别

维度 Prompt Engineering Harness Engineering
关键词 “说对话” “造环境”
改进点 提示词措辞 架构、约束、反馈
可扩展性 单次任务有效 长期运行稳定
失败处理 重新调参 重新设计环境
工程师角色 写提示词 设计系统

Harness Engineering 三大支柱

Harness 的稳定性建立在三大支柱之上：上下文工程（拒绝”说明书式”文档，采用”地图模式”）、可靠性来自”验证回路”（用结构强制它做对）、理论必须落地（实打实的生产力）。

为什么 2026 年 Harness 突然火了？

时间线：

  • 2月5日：HashiCorp 联合创始人首次定义 “Harness Engineering”
  • 2月11日：OpenAI 发布详细实验报告
  • 2月后：Martin Fowler 等权威工程师站台
  • 3月：LangChain 发布验证数据（从 52.8% 到 66.5%）

火爆原因：

  1. 所有通用大模型的核心能力越来越接近（GPT、Claude、Gemini）。当模型本身不再是差异化因素，真正的竞争力在于：谁能让 AI 更稳定、更安全、更高效地工作

  2. 当所有人都在讨论”模型更强”时，一位安全研究员仅仅改变了 Agent 的代码编辑格式就把基准得分从 6.7% 提升到 68.3%。这说明环境设计比模型选择更关键

一个生活化的类比

把 Agent 想象成是一辆车。模型是引擎，马力大、转速高。但你不能开着一台引擎上路。你需要变速箱、制动器、仪表盘、刹车。这些加在一起，任务怎么拆能顺利跑、进度怎么记、完成怎么判，这就是 Harness，就是壳。

现在你在用的 Harness

如果你使用过Claude Code，你已经在用 Harness 了——虽然你可能没意识到：

# CLAUDE.md（给 Claude 读的项目指引）
- 项目用什么技术栈
- 代码风格是什么
- 哪些文件不要动
- 怎么运行测试

这就是 Harness 的雏形——为 AI 搭建的工作环境。

关键启示

Harness Engineering 标志着 AI 辅助开发从”让模型写代码”到”设计让模型可靠工作的系统”的范式转变。瓶颈不在智能，而在基础设施。正如 Addy Osmani 所言：”AI 编码的兴起并没有取代软件工程的工艺——它抬高了工艺的门槛”。

总结一句话：

Harness Engineering 就是把 AI Agent 从野性的独角兽驯化成生产力工具的艺术。 🐴→🔧
