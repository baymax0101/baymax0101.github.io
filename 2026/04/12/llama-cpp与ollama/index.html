llama.cpp与ollama

无标签
发布日期:   2026-04-12

这两年大模型火得一塌糊涂，但是调用 OpenAI 的 API 要钱，而且还有隐私风险。越来越多的人开始研究本地部署大模型，而 llama.cpp 和 ollama 就是两个最流行的开源工具。这篇文章，我会用最通俗的方式，帮你彻底搞懂这两个东西到底是什么、有什么区别、怎么选。

1. 问题背景

先说为什么需要本地部署大模型。

第一个原因是钱。企业调用 GPT-5.4 的价格是每千个 token 3 分钱到 1 毛5，看起来不贵，但是量起来了就肉疼了。如果是个人开发者或者小团队，用量稍微大一点，每个月烧掉几千块是很正常的。

第二个原因是隐私。很多企业的数据是敏感的，不能传到云端。比如医疗数据、金融数据、法律数据，这些要是上传到 OpenAI 的服务器，轻则违反合规，重则泄露商业机密。本地部署就完全没有这个问题，数据永远留在自己的服务器上。

第三个原因是可控。你可以随便修改模型参数，可以量化压缩，可以接自己的微调模型，可以完全控制整个技术栈，不受制于任何第三方。

正是因为这些原因，本地部署大模型的需求越来越大。而 llama.cpp 和 ollama，就是这个领域最主流的两个工具。

2. 原理分析

先说 llama.cpp 是什么。

llama.cpp 是 Georgi Gerganov 开源的一个 C++ 库，专门用来运行 LLaMA 系列的模型。它的核心优势是「纯 CPU 运行」，不需要显卡也能跑模型。虽然速度比 GPU 慢，但是门槛极低，任何一台电脑都能跑。

它的原理是这样的。传统的深度学习框架比如 PyTorch、TensorFlow，主要是为 GPU 设计的，CPU 并不是它们的重点优化对象。而 llama.cpp 用了很多底层优化技巧，比如用 AVX、AVX2、AVX-512 这些 CPU 指令集来加速矩阵运算，用 quantized 量化技术把模型体积压缩到原来的几分之一甚至几十分之一。这样一来，普通电脑也能跑得动 7B、13B 的模型。

举几个具体的数字。7B 参数的 FP16 模型需要 14GB 内存，量化到 Q4 需要 4GB，量化到 Q2 需要 2GB。Q4 的模型在 M1 Mac 上大概能跑 15-20 token每秒，Q2 的模型在普通 PC 上也能跑 5-10 token每秒。这个速度虽然比 GPU 慢，但是日常使用足够了。

接下来是 ollama。

ollama 是另外一种思路的工具。它的目标是「一键运行大模型」，让你像运行一个 Docker 容器一样简单地运行大模型。它在 llama.cpp 的基础上，加了很多工程化的东西，比如模型管理、API 服务、动态显存分配、GPU 加速支持等等。

ollama 最大的优势是使用极其简单。安装好 ollama 之后，只需要一行命令就能启动一个模型：

ollama run llama2

这行命令会帮你下载模型、启动服务、创建一个交互式聊天界面，就像跟 ChatGPT 对话一样。而且 ollama 还自动暴露了一个 REST API，默认是 http://localhost:11434，你可以直接用 curl 调用：

curl http://localhost:11434/api/generate -d '{
 "model": "llama2",
 "prompt": "Write a poem about coding",
 "stream": false
}'

简单到没朋友。

总结一下两者的区别。llama.cpp 是一个底层库，给开发者用的，你可以在它的基础上做各种定制。ollama 是一个上层应用，给普通用户用的，追求的是开箱即用。如果用比喻的话，llama.cpp 像是 C 语言，ollama 像是 Python。

3. 代码实现

先看 llama.cpp 的用法。

首先克隆代码并编译：

git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
make

然后下载一个量化好的模型。你可以去 HuggingFace 找 TheBloke 做的量化模型，比如 TheBloke/LLaMA-2-7B-Chat-GGUF。下面这个命令下载 Q4 版本的模型：

# 创建模型目录
mkdir models
cd models

# 下载模型（以 7B Q4 为例）
wget https://huggingface.co/TheBloke/LLaMA-2-7B-Chat-GGUF/resolve/main/llama-2-7b-chat.Q4_K_M.gguf

接下来运行模型。llama.cpp 提供了一个叫 main 的可执行文件：

./main -m models/llama-2-7b-chat.Q4_K_M.gguf -n 256 --temp 0.7 -p "Write a python function to calculate fibonacci"

参数解释一下。-m 是模型路径，-n 是生成多少个 token，–temp 是温度参数，-p 是提示词。运行起来之后，你就可以跟模型对话了。

如果你想用 API 服务，llama.cpp 也提供了一个 server 模式：

./server -m models/llama-2-7b-chat.Q4_K_M.gguf -c 2048

默认会在 http://localhost:8080 启动一个 REST API。

再看 ollama 的用法。

安装 ollama 非常简单，Mac 和 Linux 直接一行命令：

curl -fsSL https://ollama.com/install.sh | sh

Windows 用户需要下载安装包。安装好之后，运行模型只需要一行命令：

ollama run llama2

如果你想用中文模型，可以跑阿里的中文微调版本：

ollama run qwen

ollama 也支持自定义模型。你可以把 GGUF 文件转换成 ollama 的格式，只需要一个 Modelfile：

FROM ./my-model.gguf

然后执行：

ollama create my-model -f Modelfile
ollama run my-model

如果你想启动 API 服务，ollama 默认就是以服务形式运行的。你可以在代码里面直接调用：

import requests

response = requests.post(
    "http://localhost:11434/api/generate",
    json={
        "model": "llama2",
        "prompt": "What is FastAPI?",
        "stream": False
    }
)
print(response.json()["response"])

4. 注意事项

最后说几个需要注意的点。

第一是模型的选择，并不是所有模型都适合本地运行。7B 以下的模型量化后可以在普通 PC 上跑，13B 需要 16GB 以上显存，70B 需要非常强大的显卡集群。选择模型之前，先看看自己的硬件条件。

第二是量化的选择。量化等级从高到低是 FP16、Q5_K_M、Q4_K_M、Q2_K。等级越低，模型体积越小，但是性能也越差。Q4_K_M 是性价比最高的档位，体积只有 FP16 的四分之一，质量损失很小。Q2_K 体积最小，但是明显能感觉到回答质量下降。

第三是 Windows 的支持。llama.cpp 对 Windows 的支持不如 Mac 和 Linux 好，编译和运行都可能遇到各种奇怪的问题。如果你是 Windows 用户，建议直接用 ollama，它对 Windows 的支持比较完善。

第四是交互体验。纯 CPU 运行的模型，响应速度是硬伤。你需要耐心等待模型输出，体验远远不如云端模型。如果你想追求更好的体验，可以考虑买一块大显存显卡，比如 RTX 5080 或者 5090，这能让本地大模型的体验提升一个档次。
